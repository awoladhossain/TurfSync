# Redis

## RedisModule

1. `@Global() ডেকোরেটর: কেন এবং কখন?`
   আপনি এখানে @Global() ব্যবহার করেছেন। এর মানে হলো, এই মডিউলটি একবার AppModule-এ ইমপোর্ট করলে পুরো অ্যাপ্লিকেশনের যেকোনো জায়গায় (Booking, Turf, User মডিউল) আপনি Redis ব্যবহার করতে পারবেন। বারবার ইমপোর্ট করার ঝামেলা নেই।

   Deep Insight: বড় সিস্টেমে গ্লোবাল মডিউল সাবধানে ব্যবহার করতে হয় যাতে ডিপেন্ডেন্সি গ্রাফ জটিল না হয়। তবে Redis-এর মতো কোর সার্ভিস (Shared Utility) সাধারণত গ্লোবাল হওয়াই স্ট্যান্ডার্ড।

2. `REDIS_CLIENT এবং Injection Token`

```bash
provide: REDIS_CLIENT,
```

আপনি সরাসরি Redis ক্লাস ব্যবহার না করে একটি কনস্ট্যান্ট টোকেন (REDIS_CLIENT) ব্যবহার করেছেন। এটি একটি Best Practice।
কেন? যদি ভবিষ্যতে আপনি ioredis লাইব্রেরি বদলে অন্য কোনো লাইব্রেরি ব্যবহার করতে চান, তবে আপনাকে পুরো কোডবেস হাত দিতে হবে না। শুধু এই প্রোভাইডারে চেঞ্জ করলেই হবে। একে বলা হয় Decoupling।

3. `useFactory এবং ডাইনামিক কনফিগারেশন`

```bash
useFactory: (configService: ConfigService) => { ... }
```

এটি একটি "ফ্যাক্টরি ডিজাইন প্যাটার্ন"। আপনি অ্যাপ রান হওয়ার সময় সিদ্ধান্ত নিচ্ছেন Redis কোথায় কানেক্ট হবে।
ConfigService: সরাসরি process.env ব্যবহার না করে ConfigService ব্যবহার করা হয়েছে, যা NestJS-এর স্ট্যান্ডার্ড। এটি নিশ্চিত করে যে এনভায়রনমেন্ট ভেরিয়েবলগুলো টাইপ-সেফ এবং ভ্যালিডেটেড।

4. `Retry Strategy: হার্ট অফ কানেকশন (Deep Knowledge)`

```bash
retryStrategy: (times) => Math.min(times * 50, 2000),
```

এই ছোট লাইনটি আপনার অ্যাপ্লিকেশনের Resilience বা টিকে থাকার ক্ষমতা নির্ধারণ করে। অনেক ইঞ্জিনিয়ার এটি দেন না, যার ফলে Redis সার্ভার একবার রিস্টার্ট হলে অ্যাপ ক্রাশ করে বা কানেকশন ফিরে পায় না।

- লজিক: times হলো কতবার কানেকশন ট্রাই করা হয়েছে।

- ১ম বার: ৫০ মিলি-সেকেন্ড পর ট্রাই করবে।

- ১০ম বার: ৫০০ মিলি-সেকেন্ড পর।

- Max Cap: এটি কখনোই ২০০০ মিলি-সেকেন্ড (২ সেকেন্ড) এর বেশি দেরি করবে না (Math.min এর কারণে)।

কেন এটি দরকার? একে বলা হয় `Exponential Backoff`। যখন Redis সার্ভার ডাউন থাকে, তখন যদি আপনার অ্যাপ প্রতি মিলি-সেকেন্ডে রিকোয়েস্ট পাঠাতে থাকে, তবে সার্ভার ঠিক হওয়ার সময় কয়েক হাজার কানেকশন রিকোয়েস্টের চাপে আবার ডাউন হয়ে যেতে পারে `(Thundering Herd Problem)`। এই স্ট্র্যাটেজি সেটি থামায়।

5. `RedisLockService: Concurrency হ্যান্ডেল করার অস্ত্র`

আপনি এই মডিউলে `RedisLockService` ও রেখেছেন। টার্ফ বুকিংয়ের মতো সিস্টেমে এটিই সবচেয়ে বড় ভূমিকা পালন করবে।

- Scenario: ধরুন, রাত ৯টার স্লট একটিই খালি আছে। দুইজন ইউজার একই মাইক্রো-সেকেন্ডে "Book Now" বাটনে চাপ দিলেন।

- Solution: RedisLockService একটি Distributed Lock তৈরি করবে।s
- ইউজার ১ একটি চাবি (Lock) পাবে এবং ডাটাবেজে বুকিং শুরু করবে।
- ইউজার ২-কে Redis বলবে "Wait, this resource is locked"।
- ইউজার ১-এর কাজ শেষ হলে চাবি রিলিজ হবে।

## 🏗️ RedisLockService: Deep Dive Analysis

এই সার্ভিসটি মূলত দুটি কাজ করছে:

1. Distributed Locking: কনকারেন্সি বা ডাবল বুকিং আটকানো।
2. Utility Wrapper: Redis-এর জটিল কমান্ডগুলোকে সহজ করে দিচ্ছে।

### Deep dive

1. `acquireLock`

```bash
async acquireLock(key: string, ttlSeconds = 30): Promise<boolean> {
  const result = await this.redis.set(`lock: ${key}`, '1', 'EX', ttlSeconds, 'NX');
  return result === 'OK';
}
```

Deep Logic: এখানে ৩টি বিশেষ প্যারামিটার ব্যবহার করা হয়েছে:

- NX (Not Exists): এই কমান্ডটি তখনই সাকসেস হবে যদি Redis-এ ওই key আগে থেকে না থাকে। যদি দুইজন ইউজার একসাথে বুক করতে চায়, তবে প্রথমজন OK পাবে, আর দ্বিতীয়জন ব্যর্থ হবে কারণ কী-টি অলরেডি তৈরি হয়ে আছে।
- EX (Expire): আপনি লকটি ৩০ সেকেন্ডের জন্য নিচ্ছেন।

- কেন ৩০ সেকেন্ড? ধরুন কোনো কারণে আপনার বুকিং প্রসেসের মাঝে সার্ভার ক্রাশ করল। লকটি যদি অটো-এক্সপায়ার না হতো, তবে ওই স্লটটি সারাজীবনের জন্য ব্লক হয়ে যেত। একে বলা হয় Deadlock Prevention।

2. এই key টা আসবে কোথা থেকে?
   এই key টা মূলত আপনি নিজে ডিফাইন করবেন যখন আপনি সার্ভিসটি কল করবেন। এটি এমন কিছু হওয়া উচিত যা একটি নির্দিষ্ট রিসোর্সকে ইউনিকলি রিপ্রেজেন্ট করে।
   টার্ফ বুকিংয়ের ক্ষেত্রে: ধরুন আপনি একটি নির্দিষ্ট টার্ফের একটি নির্দিষ্ট স্লট বুক করছেন। তখন কি-টা হতে পারে: `turf:123:slot:2026-04-30:10pm।`

```bash
const lockKey = `booking:${turfId}:${slotId}`;
const isLocked = await this.redisLockService.acquireLock(lockKey);
```

অর্থাৎ, আপনি যে জিনিসটাকে "লক" করতে চান, তার একটা নামই হলো এই key।

3. এখানে '1' এর মানে কী?
   Redis-এর SET কমান্ডে আপনাকে একটি ভ্যালু দিতেই হয়। এখানে '1' দেওয়া মানে হলো আমরা জাস্ট একটা "ফ্ল্যাগ" বা চিহ্ন রাখছি যে এই কী-টি এখন অকুপাইড বা বুকড।

- লজিক: এখানে আপনি '1' এর বদলে 'locked', 'true' বা আপনার নামও লিখতে পারতেন, তাতে লজিকে কোনো পরিবর্তন আসতো না।
- গুরুত্ব: আমাদের আসল উদ্দেশ্য ভ্যালু কী তা দেখা নয়, বরং Redis-এ ওই key টা বিদ্যমান আছে কি না তা দেখা (NX অপশনের মাধ্যমে)।

### প্রসেস ফ্লো (Step-by-Step)

1. রিকোয়েস্ট: ইউজার একটি কাজের জন্য লক চায়।
2. চেকিং: acquireLock চেক করে Redis-এ ওই key আছে কি না।
3. ডিসিশন:

- যদি না থাকে → লক সেট হয়, ফাংশন true রিটার্ন করে এবং ইউজার কাজ শুরু করে।
- যদি থাকে → ফাংশন false রিটার্ন করে এবং ইউজারকে অপেক্ষা করতে বলা হয়।

4. রিলিজ: কাজ শেষ হলে releaseLock কল করে চাবিটি ফেলে দেওয়া হয় যাতে অন্যরা সুযোগ পায়।
   এটি মূলত ডিস্ট্রিবিউটেড সিস্টেমে Data Integrity বা তথ্যের সঠিকতা বজায় রাখার একটি স্ট্যান্ডার্ড পদ্ধতি।
